# Model-Native Knowledge and Explicit Provisioning

Assume no authoritative knowledge, data, connector, workflow, policy, specialist, model, handler, execution capability, or proof exists unless it is physically present, registered, configured, authorized, tested, deployed, verified, and linked to proof.

Model-native knowledge may support general reasoning, explanation, drafting, and candidate generation. It is provider-dependent, cutoff-limited, probabilistic, and not proof of current facts, organization-specific facts, runtime capability, or completed work.

Use these source classifications where knowledge provenance is represented:

- `model_native`
- `platform_knowledge`
- `tenant_knowledge`
- `retrieved_evidence`
- `live_external_source`
- `runtime_evidence`

Generated knowledge, code, schemas, policies, workflows, prompts, specialist definitions, tests, and documentation are candidates until reviewed, tested or evaluated, approved, versioned, registered, deployed, and verified.

The versioned manifest at `config/platform-assets.v1.json` covers knowledge, data, schemas, handlers, connectors, workflows, specialists, prompts, indexes, models, policies, evaluations, APIs, UI modules, proofs, receipts, and test fixtures. Each entry records provenance, verification state, proof references, and limitations.
